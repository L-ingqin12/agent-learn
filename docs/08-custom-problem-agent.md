# 第八部分：定制化问题分析 Agent 开发

## 1. 问题分析 Agent 的本质

大多数定制化 Agent 的本质是**诊断型智能体**：给定一个现象，找出根因，给出方案。

```
用户报告问题 → Agent 收集证据 → 形成假设 → 验证假设 → 输出诊断报告
```

这与医生诊断、工程师排查故障、安全审计是同一套认知模式。

---

## 2. 通用分析协议：O-H-V-C

无论分析什么问题，都遵循四步法：

| 阶段 | 英文 | 动作 | 产物 |
|------|------|------|------|
| **O**bserve | 观察 | 收集症状、日志、数据 | 证据清单 |
| **H**ypothesize | 假设 | 根据领域知识推断可能原因 | 假设列表（排序） |
| **V**erify | 验证 | 设计测试/查询来证伪每个假设 | 验证结果 |
| **C**onclude | 结论 | 确定根因，给出修复方案 | 诊断报告 |

### 为什么是"证伪"而非"证实"？

确认偏误是诊断的敌人。好的分析 Agent 应尝试**排除**每个假设：
- "如果假设 X 成立，我应该能看到证据 Y。Y 存在吗？"
- 排除所有不成立的假设后，剩下的就是根因。

---

## 3. 领域知识的三层编码

定制化 Agent 的核心区别在于**领域知识**。将知识分三层编码：

### 第一层：故障模式库 (Failure Mode Catalog)

```
FailureMode = {
    name: str           # 故障名称
    symptoms: list      # 可观测症状
    preconditions: list # 发生前提
    severity: enum      # 严重程度
    evidence: list      # 确诊所需证据
    fix: str            # 修复方案
}
```

### 第二层：诊断规则 (Diagnostic Rules)

```
Rule = {
    if: 条件（证据组合）
    then: 结论（指向某个 FailureMode）
    confidence: 置信度
}
```

### 第三层：证据收集策略 (Evidence Collection Strategy)

```
Strategy = {
    target: FailureMode     # 针对哪个故障模式
    tools: list             # 使用哪些工具
    queries: list           # 执行哪些查询
    expected: dict          # 期望看到什么（如果是该故障）
}
```

## 4. 分析流程设计模式

### 4.1 分层递进式 (Triage Pattern)

```
第一层: 快速分类（排除 80% 的可能）
    → 检查最明显的信号
    → 窄化到 2-3 个候选
第二层: 定向取证
    → 针对候选收集更深证据
第三层: 确认诊断
    → 验证唯一假设
    → 输出方案
```

### 4.2 决策树式 (Decision Tree Pattern)

```
症状A?
  ├─ Yes → 检查B
  │        ├─ Yes → 诊断X
  │        └─ No  → 检查C
  └─ No  → 检查D
           ├─ Yes → 诊断Y
           └─ No  → 诊断Z
```

### 4.3 差分诊断式 (Differential Diagnosis Pattern)

```
列出所有可能的故障模式
  → 按概率排序
  → 逐个设计"排除测试"
  → 排除后剩余的 = 根因
```

## 5. 定制化开发步骤

### Step 1: 定义问题域

```
目标问题类型: 代码Bug / 性能问题 / 安全漏洞 / 数据异常 / 运维故障
分析深度: 自动修复 / 辅助诊断 / 根因分析 / 影响评估
输出格式: 诊断报告 / 修复方案 / 风险评估 / Action Items
```

### Step 2: 梳理领域知识

```
常见故障模式有哪些？
每种故障的典型症状是什么？
需要哪些证据来确认/排除？
修复方案是什么？
```

### Step 3: 设计证据收集工具

```
需要读取什么数据？（文件、日志、数据库、API）
需要运行什么检查？（测试、脚本、查询）
需要问用户什么问题？（澄清信息收集）
```

### Step 4: 设计分析工作流

```
选择分析模式（分层/决策树/差分诊断）
确定步骤顺序和条件分支
确定何时停止（置信度 / 步骤上限）
```

### Step 5: 实现并验证

```
用已知案例测试
收集假阳性/假阴性
迭代优化规则
```

---

## 6. 设计原则

1. **先排除再确认** — 不要找证据"证明"假设，找证据"排除"假设
2. **信息不够时主动问** — 不要猜测，向用户请求澄清
3. **分层渐进** — 先看便宜的信号，再收集贵的证据
4. **可解释** — 每步输出推理过程，让用户信任诊断
5. **优雅降级** — 找不到根因时，至少给出排查方向和建议
6. **领域知识与推理引擎分离** — 换一个问题域只需换知识库，不改流程
