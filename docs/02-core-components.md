# 第二部分：Agent 核心组件详解

## 2.1 LLM 推理引擎

LLM 是 Agent 的大脑，负责理解任务、分解步骤、决定何时调用工具。

### 关键能力要求
- **指令遵循（Instruction Following）** — 严格按照 system prompt 执行
- **函数调用（Function Calling / Tool Use）** — 准确选择工具和参数
- **多步推理（Chain-of-Thought）** — 分解复杂问题
- **上下文管理（Context Window）** — 在有限窗口内高效工作

### 模型选择建议

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 复杂推理任务 | Claude Opus | 深度推理能力最强 |
| 高吞吐场景 | Claude Sonnet | 性价比最优 |
| 简单分类/路由 | Claude Haiku | 低延迟低成本 |

## 2.2 工具系统（Tools）

工具是 Agent 与外部世界交互的接口，让 Agent 能够执行 LLM 本身无法完成的操作。

### 工具的类型
- **API 工具** — 调用外部服务（数据库查询、HTTP 请求）
- **文件工具** — 读写文件
- **代码执行** — 运行脚本进行计算
- **搜索工具** — 检索外部知识
- **UI 工具** — 操作浏览器或桌面应用

### 工具定义的关键要素

```
{
  "name": "search_database",
  "description": "搜索用户数据库，输入查询条件，返回匹配的用户列表",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": { "type": "string", "description": "搜索关键词" },
      "limit": { "type": "integer", "description": "返回数量上限" }
    },
    "required": ["query"]
  }
}
```

### 设计原则
1. **单一职责** — 每个工具只做一件事
2. **描述清晰** — description 决定 LLM 能否正确选择工具
3. **参数明确** — schema 定义精确，减少调用错误
4. **错误友好** — 返回有意义的错误信息，帮助 Agent 自我纠正

## 2.3 记忆系统（Memory）

记忆让 Agent 能够跨轮次保持上下文，做出连续性的决策。

### 记忆层次

| 层次 | 时效 | 用途 | 实现方式 |
|------|------|------|---------|
| 工作记忆 | 单次对话 | 当前任务上下文 | 对话历史、context window |
| 短期记忆 | 数分钟-数小时 | 会话内信息 | Redis、内存缓存 |
| 长期记忆 | 天-月-年 | 用户偏好、知识积累 | 向量数据库、知识图谱 |

### 记忆管理策略
- **摘要压缩** — 将长对话压缩为摘要放入 context
- **向量检索** — 语义搜索相关历史信息
- **关键信息提取** — 只保留决策和偏好，丢弃闲聊

## 2.4 规划系统（Planning）

规划能力决定了 Agent 能否完成复杂的多步任务。

### 规划方法

1. **ReAct 模式**
   - 交替进行 "思考→行动→观察" 循环
   - 灵活应对动态变化

2. **Plan-and-Solve**
   - 先制定完整计划
   - 逐步执行每个步骤
   - 适合步骤明确的确定性任务

3. **分层规划**
   - 高层规划定方向
   - 低层规划定动作
   - 适合超长任务

### 规划中的常见问题
- **过度规划** — 对简单任务制定复杂计划
- **计划僵化** — 无法根据执行反馈调整计划
- **目标漂移** — 在执行中偏离原始目标

## 2.5 错误处理与反思

优秀的 Agent 需要能识别错误并自我纠正。

### 反思机制
- **结果验证** — 检查工具调用结果是否符合预期
- **错误重试** — 分析错误原因后重新尝试
- **降级策略** — 当前方法失败时使用备选方案
- **请求澄清** — 信息不足时向用户确认
